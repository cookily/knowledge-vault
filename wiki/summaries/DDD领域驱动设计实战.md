# 从Alibaba-Cola到DDD，一线研发对领域驱动的思考

## 一句话总结

DDD 落地核心是**知道代码放哪层**。Alibaba-Cola 提供了标准分层模板：Client → Adapter → App → Domain → Infra。

## Cola 架构分层

```
┌─────────────────────────────────────────┐
│              client 层（可选）             │  提供对外接口的 SDK
├─────────────────────────────────────────┤
│              adapter 层                  │  Controller、参数校验、定时任务
├─────────────────────────────────────────┤
│              app 层                      │  协调者：组装上下文、调用领域层
├─────────────────────────────────────────┤
│              domain 层 ★核心             │  业务逻辑、不依赖其他层
├─────────────────────────────────────────┤
│              infra 层                    │  技术实现：MyBatis、MQ、配置类
└─────────────────────────────────────────┘
```

## Domain 层核心要素

| 要素 | 说明 |
|------|------|
| **实体** | 有 ID、生命周期、状态（如工单） |
| **值对象** | 无 ID、不可变、可替换（如地址） |
| **聚合** | 多个实体+值对象组成，具备行为 |
| **工厂** | 创建复杂聚合体 |
| **服务接口** | 领域行为，无状态 |
| **仓储接口** | 定义持久化能力（实现交给 Infra） |

## Domain 层三大原则

1. **不依赖其他任何模块**
2. **业务逻辑脱离基础设施**（不依赖 MySQL/Oracle）
3. **不是贫血模型**，要有行为

## 实战：工单系统建模

**实体**：工单（WorkOrder）
- ID：唯一标识
- 生命周期：INITIAL → HANDLEING → COMPLETED
- 状态：可驳回重新处理

**值对象**：
- 工单内容（图片、类型、文字）
- 工单状态（枚举）
- 处理结果（处理人、时间、意见）

**关键方法**：
```java
WorkOrder.start()    // 发起工单
workOrder.handle()   // 处理工单
workOrder.finish()   // 完成
workOrder.reject()   // 驳回（创建子工单）
```

## 核心观点

> 没有最好的架构。如果能通过合理编排层次实现代码解耦、易读，就是好设计。

- 项目不大，不需要套用完整四层
- 不需要用所有 DDD 概念（如本文没用到「工厂」）
- 实体直接当聚合体即可，徒增复杂度没必要
