# 摘要：UI-TARS Desktop - 字节跳动开源看屏幕操作电脑 AI Agent

## 一句话总结

UI-TARS 是字节跳动开源的"视觉操控"AI Agent：不调 API、不读 DOM，直接看截图识别按钮位置然后模拟鼠标键盘操作，像真人远程操控电脑，GitHub 34K+ stars。

## 核心内容

### 工作原理（截屏 → 看图 → 决策 → 执行 → 循环）

1. **截屏**：实时捕获当前屏幕画面
2. **视觉语言模型**：看图理解当前界面状态（识别按钮、输入框、菜单等）
3. **决策**：根据任务目标决定点哪、输入什么、滚动多少
4. **执行**：模拟鼠标点击、键盘输入
5. **验证**：再截屏看结果，循环直到任务完成

**核心区别**：不依赖任何 API 或 DOM 结构，纯视觉驱动。传统方案（Playwright/Puppeteer）需要开发者写脚本指定元素位置；UI-TARS 只需要用自然语言描述任务，它自己找界面上的操作入口。

### 典型 Demo

- 打开浏览器 → 搜索航班信息 → 填表 → 切日历 app 标记日期 → 回到浏览器下单
- 全程没有调用任何 API，没有读 DOM —— 就是看着屏幕截图操作

### 技术定位

| 对比维度 | UI-TARS | 传统浏览器自动化 | API 调用 |
|----------|---------|-----------------|----------|
| 技术路线 | 视觉感知 | DOM 定位 | HTTP 请求 |
| 操作方式 | 自然语言描述 | 写脚本找元素 | 预设接口 |
| 适用场景 | 通用界面操作 | 结构化网页 | 开放 API 的服务 |
| 维护成本 | 低（界面变了他自己适应） | 高（页面改版脚本就坏） | 中 |

### 适用场景

- 重复性桌面操作（填表、截图、文件整理）
- 无 API 的内部系统自动化
- 需要跨应用切换的工作流（浏览器→桌面 app→浏览器）
- 测试场景（端到端 UI 测试）

### 局限与风险

- 视觉识别有误差率，复杂任务需人工确认
- 安全风险：可操作任何界面，需在隔离环境使用
- 不适合精确数值输入（如精确坐标点击）
