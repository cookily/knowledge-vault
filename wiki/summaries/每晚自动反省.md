# 摘要：每晚自动反省——让你的 Hermes 加速进化

## 一句话总结

给 Hermes 部署每日凌晨自动巡检任务，揪出"服务活着但悄悄出错"的隐性问题（如代理过期、模型端点 404、LiteLLM 组内故障），从"纠错日记"升级为"系统巡检"，同时强化自我进化能力。

## 核心问题

**"服务活着，但一直在悄悄出错"的隐秘问题，光靠人工根本发现不了。**

典型场景：
- 代理订阅 token 过期 → 更新脚本拿到空内容 → 配置文件停在三天前旧版本（服务没挂，但规则已过期）
- LiteLLM 网关某模型组，前三个端点间歇性 404，第四个兜底成功 → 主对话不受影响，但前三个实际已经失败
- 向量数据库高频整理后被"算死" → 记忆丢失

## Hermes 自带的进化机制

Hermes 官方定位"the self-improving AI agent"，内置四套自动学习机制：

| 机制 | 触发条件 | 功能 |
|------|----------|------|
| **Memory Nudge** | 每 10 轮对话 | 自动判断值得长期保存的用户偏好/环境信息 |
| **Skill Nudge** | 每 10 次工具调用 | 将操作提炼为可复用技能 |
| **Background Review Fork** | 对话过程中 | spawn 后台线程，aux 模型审查对话提取知识 |
| **Curator（技能管家）** | 每 7 天 | 扫描技能库，归档 30 天未用技能，合并重叠技能 |

## 设计理念：巡检不是找犯错

**纠错日记**（v1）→ 每天翻聊天记录看犯了什么错

问题：定位消极，主动性不足

**系统巡检**（v3）→ 凌晨定时跑，检查后台服务状态，揪隐性问题

升级：从被动纠错 → 主动防御，让 Hermes 更懂用户、更稳定

## 每日巡检任务设计

### 巡检范围

- **LiteLLM 网关**：模型端点可用性、响应时间、错误率
- **代理/mihomo**：订阅有效性、配置版本
- **OpenViking（记忆存储）**：写入一致性
- **自动备份脚本**：最近备份时间、备份完整性
- **各模型组**：组内故障转移是否被触发过

### 工具

- **Cron no_agent 模式**（v0.13+）：纯脚本定时任务，不走 Agent，适合健康检查
- **context_from**：上一个 job 输出作为下一个 job 上下文，实现任务链
- **LLM 辅助分析**：巡检结果用 LLM 总结，输出人类可读的巡检报告

### 巡检报告示例

```
=== Hermes 系统巡检报告 ===
时间：2026-05-17 02:00

[✓] LiteLLM 网关：正常，4/4 模型响应 < 2s
[!] mihomo 代理：警告 - 订阅配置与昨天相同，建议刷新
[✓] OpenViking 记忆：正常，昨晚写入 3 条
[✓] 自动备份：正常，备份时间 01:30，大小 2.3GB
```

## 复用方法

1. **创建 Cron Job**：凌晨 2:00 触发，使用 `no_agent` 模式
2. **脚本设计**：ping 各服务 → 记录结果 → LLM 总结
3. **报告存储**：结果写入指定文件，供 Hermes 第二天读取
4. **持续迭代**：根据暴露的问题调整巡检规则
